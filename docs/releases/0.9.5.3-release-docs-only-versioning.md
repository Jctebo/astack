## Scope

### Problem
- astack still has two competing release systems.
- The workflow and archival process already use `docs/releases/VERSION`, `docs/releases/RELEASE_LOG.md`, and versioned release artifacts.
- At the same time, updater, setup, telemetry, upgrade summaries, and some docs still depend on repo-root `VERSION` and `CHANGELOG.md`.
- That split creates ambiguity about which version is canonical and makes the release process feel only partially migrated.

### Audience
- Primary user: astack maintainers and contributors who ship new astack versions and need one obvious release contract.
- Secondary user: astack users running update and upgrade flows who expect the reported version and release notes to match the shipped archival docs.

### Current Status Quo
- `docs/releases/` is the canonical workflow archive for scope, research, plan, progress, QA, and release notes.
- Repo-root `VERSION` and `CHANGELOG.md` still exist and are used by:
- update checks
- telemetry logging
- setup/runtime-root metadata copying
- upgrade summaries
- tests and docs that assume a top-level version file
- As a result, astack can describe `docs/releases/` as canonical while still reading machine-facing release metadata from repo root.

### Goal
- Make `docs/releases/` the single source of truth for astack versioning and release history.
- Remove repo-root `VERSION` and `CHANGELOG.md` from the runtime, docs, tests, and upgrade flow.
- Ensure updater, setup, telemetry, and upgrade summaries all read from `docs/releases/VERSION` and `docs/releases/RELEASE_LOG.md`.

### Constraints
- Update checks must still work for installed runtime roots that are slim copies rather than full source checkouts.
- Upgrade flows must still be able to summarize what changed between versions after the root changelog is removed.
- The release system must stay host-agnostic across Claude, Codex, and Copilot installs.
- Existing shipped versions and tests may still assume root-level files.

### Narrow Wedge
- Convert all astack-owned code paths that read or copy root `VERSION` / `CHANGELOG.md` to the release-folder equivalents.
- Delete the root files once all internal references are migrated.
- Update generated skill text, setup behavior, tests, and docs in the same pass so there is no split contract left behind.

### Non-Goals
- Changing the four-digit version format.
- Redesigning the updater UX or adding a new remote service.
- Migrating external target repos beyond the astack release contract they consume.

### Risks And Unknowns
- Remote update checks currently fetch the raw top-level `VERSION` file from GitHub; changing that path must be done carefully.
- Installed runtime roots may need release metadata copied into a nested `docs/releases/` path instead of only top-level files.
- Some older generated host outputs may still mention root files even after the source templates change.

### Recommended Next Step
- Use `/research-astack` to map every remaining dependency on root `VERSION` and `CHANGELOG.md`, then use `/plan-astack` to lock the migration path before implementation.

## Research

### Scope Baseline
- This release converges astack's machine-facing version and release history onto the same `docs/releases/` contract that already powers workflow archival.

### Relevant Files
- `bin/astack-update-check`
  Why it matters: currently reads local version from `$ASTACK_DIR/VERSION` and fetches the remote version from the repo-root `VERSION` URL.
- `bin/astack-telemetry-log`
  Why it matters: currently logs `astack_version` from `$ASTACK_DIR/VERSION`.
- `setup`
  Why it matters: copies root `VERSION` and `CHANGELOG.md` into slim runtime roots for installed hosts.
- `astack-upgrade/SKILL.md.tmpl`
  Why it matters: upgrade flow reads installed `VERSION`, writes `just-upgraded-from`, compares local vs vendored versions, and summarizes changes from `CHANGELOG.md`.
- `browse/test/astack-update-check.test.ts`
  Why it matters: large regression suite assumes `VERSION` lives at the install root and remote version is fetched from a single raw `VERSION` file.
- `README.md`, `AGENTS.md`, `ARCHITECTURE.md`, `CLAUDE.md`, `docs/host-support.md`
  Why they matter: host and workflow docs must describe one canonical release contract.
- `test/skill-validation.test.ts`, `test/gen-skill-docs.test.ts`, `test/skill-e2e.test.ts`, `test/skill-routing-e2e.test.ts`
  Why they matter: they enforce generated wording and release-folder expectations.

### Current System Map
- Workflow planning and shipping already use:
  - `docs/releases/VERSION`
  - `docs/releases/RELEASE_LOG.md`
  - `docs/releases/<version>-<slug>.md`
- Runtime/update paths still use repo-root release files:
  - `bin/astack-update-check` reads `$ASTACK_DIR/VERSION`
  - the default remote check URL is `https://raw.githubusercontent.com/Jctebo/astack/main/VERSION`
  - `bin/astack-telemetry-log` reads `$ASTACK_DIR/VERSION`
  - `setup` copies `$ASTACK_DIR/VERSION` and `$ASTACK_DIR/CHANGELOG.md` into runtime roots
  - `astack-upgrade/SKILL.md.tmpl` reads `${SOURCE_DIR:-$INSTALL_DIR}/VERSION` and `${SOURCE_DIR:-$INSTALL_DIR}/CHANGELOG.md`
- Installed runtime roots are intentionally slim, so they cannot rely on the source checkout being present for release metadata.
- The current shipped release log already lives at `docs/releases/RELEASE_LOG.md`, which means the root `CHANGELOG.md` is now redundant rather than canonical.

### Reuse Opportunities
- Reuse `docs/releases/VERSION` directly as the canonical local version source.
- Reuse `docs/releases/RELEASE_LOG.md` directly as the canonical release-history source for upgrade summaries.
- Reuse the existing `.astack-source-path` metadata model in runtime roots rather than inventing another upgrade lookup mechanism.
- Reuse the release artifact flow already adopted by ship/document-release so the human-facing and machine-facing release contracts finally match.

### Constraints
- Update checks need a raw GitHub URL that returns only the version string, so the remote path must change from `/VERSION` to `/docs/releases/VERSION`.
- Slim runtime roots need nested release metadata copied locally so update checks and upgrade summaries still work without a full source checkout.
- Upgrade summaries currently rely on parsing `CHANGELOG.md` version headings; the parser behavior must stay compatible with `docs/releases/RELEASE_LOG.md`.
- Tests cover both isolated temp installs and the real project root, so they must stop assuming root files exist.

### Risks
- Highest risk: update checks silently stop working if either the local runtime layout or the remote raw URL is changed incompletely.
- High risk: upgrade flows keep reading stale top-level paths while setup starts copying only release-folder files.
- High risk: generated skill outputs drift from source templates if regeneration is missed after template updates.
- Medium risk: documentation still mentions root `CHANGELOG.md` or `VERSION`, leaving users with an outdated manual upgrade model.

### Assumptions To Validate
- `docs/releases/RELEASE_LOG.md` has a heading format stable enough to replace `CHANGELOG.md` for upgrade summaries.
- Runtime roots can safely carry a nested `docs/releases/` subtree without affecting host skill discovery.
- No external tooling outside this repo still requires the root `VERSION` or `CHANGELOG.md` files once astack's own code is migrated.

### Unknowns
- Whether any generated host install or third-party consumer reads the root files directly outside the tested paths.
- Whether older local vendored copies upgraded from earlier releases will need fallback logic for missing `docs/releases/` metadata.

### Recommended Direction
- Treat `docs/releases/` as the only release contract and migrate every astack-owned consumer in one pass.
- Update runtime metadata copying so installed roots contain:
  - `docs/releases/VERSION`
  - `docs/releases/RELEASE_LOG.md`
- Update update-check, telemetry, and upgrade flows to read the release-folder paths first and stop depending on root files.
- Delete repo-root `VERSION` and `CHANGELOG.md` after tests, docs, and generated outputs are aligned.

## Plan

### Summary
- Converge astack's release source of truth onto `docs/releases/` only.
- Remove repo-root `VERSION` and `CHANGELOG.md` from the repo and from all astack-owned runtime flows.
- Keep update, upgrade, and telemetry behavior intact by teaching runtime roots and scripts to read `docs/releases/VERSION` and `docs/releases/RELEASE_LOG.md`.

### What Already Exists
- `docs/releases/VERSION` already tracks astack's shipped version cadence.
- `docs/releases/RELEASE_LOG.md` already contains shipped release notes in the correct historical format.
- `setup` already copies release metadata into runtime roots; it just copies the wrong file locations today.
- Generated skill docs are derived from source templates, so one coordinated template change can update all hosts.

### Architecture
- Canonical release metadata lives only under `docs/releases/`.
- Installed runtime roots should include a nested `docs/releases/` directory containing:
  - `VERSION`
  - `RELEASE_LOG.md`
- Runtime scripts will resolve release metadata from `$ASTACK_DIR/docs/releases/...` instead of `$ASTACK_DIR/...`.
- Remote update checks will fetch `https://raw.githubusercontent.com/<repo>/main/docs/releases/VERSION`.
- Upgrade summaries will read `docs/releases/RELEASE_LOG.md` instead of `CHANGELOG.md`.
- The root release files will be deleted once every internal reader is migrated.

### Failure Modes
- Local install metadata is copied to the wrong place, causing update checks to silently skip because `docs/releases/VERSION` is missing in runtime roots.
- The remote raw URL is changed locally but not reflected in tests, hiding a broken release check until after ship.
- Upgrade summaries fail because the release log parser expects the old changelog format.
- Root-file references remain in generated skills or docs and reintroduce split-contract confusion.

### QA And Test Matrix
- Update check behavior:
  - missing `docs/releases/VERSION`
  - empty `docs/releases/VERSION`
  - matching local and remote versions
  - differing local and remote versions
  - stale and fresh cache paths
  - force refresh path
- Upgrade flow behavior:
  - reads old version from release-folder metadata
  - compares local vendored versions using release-folder metadata
  - summarizes shipped changes from `docs/releases/RELEASE_LOG.md`
- Setup/runtime behavior:
  - runtime roots receive nested `docs/releases/` metadata
  - legacy source-path metadata remains intact
- Documentation correctness:
  - no live docs claim repo-root `VERSION` or `CHANGELOG.md` are canonical

### Rollout Notes
- This should be a single-pass cutover on the feature branch.
- After ship, the old root files should be gone rather than left as compatibility mirrors.

### Not In Scope
- Changing the release artifact filename scheme.
- Reworking Supabase's edge function version default in this same pass unless tests show it is tied to the root file migration.
