## Scope

### Problem
- A single enhancement still requires too much manual coordination across scope, research, planning, implementation, QA, shipping, docs sync, and branch cleanup.
- The existing workflow can guide each stage, but a user still has to remember to invoke the right skills in the right order and keep track of branch state.
- We want one unattended request to be able to carry an enhancement from the first scope pass all the way to a shipped PR, while still stopping for any decision that needs a human.

### Audience
- Primary user: astack maintainers and contributors who want one enhancement request to run end to end with minimal supervision.
- Secondary stakeholders: reviewers and teammates who benefit from consistent branch handling, release artifacts, and clean post-merge state.

### Current Status Quo
- Scope, research, plan, implement, QA, and ship are available as separate skills.
- Branch bootstrap already happens when the planning flow starts on `main`.
- Ship already handles PR creation, release docs sync, merge prompting, and local cleanup after merge.
- None of those steps are yet orchestrated as one unattended enhancement run.

### Goal
- Let one enhancement request execute the full astack path from scope through ship without manual step-by-step orchestration.
- Reuse the existing skill boundaries instead of folding the workflow into one monolithic command.
- Retry clearly transient failures once, then stop and surface the failure if it is not obviously recoverable.
- Preserve the existing safety gates around human judgment, merge decisions, dirty trees, and ambiguous failures.

### Constraints
- Must continue to use the existing release artifact as the durable source of truth between stages.
- Must respect the enhancement branch lifecycle and keep branch bootstrap deterministic from the release slug.
- Must not hide failures that require a user decision.
- Must keep QA and ship behavior compatible with the existing review and release workflow.

### Narrow Wedge
- Automate the happy path for a single enhancement request:
  - scope
  - research
  - plan
  - implement
  - QA
  - ship
  - docs sync
  - merge prompt
  - local cleanup after merge
- Keep retry logic narrow: one retry for transient command/test failures, no retries for branch conflicts or human decisions.

### Non-Goals
- Fully autonomous recovery from ambiguous failures.
- Auto-merging without user confirmation.
- Parallel multi-enhancement execution.
- Replacing the existing skill workflows with a single new hidden engine.

### Risks And Unknowns
- The biggest risk is over-automation: the workflow could keep going when it should stop and ask for help.
- Retry handling needs to distinguish transient failures from branch or merge problems.
- We need to confirm which orchestration boundaries belong in the new enhancement versus staying inside the existing individual skills.
- The unattended path must still leave a sensible branch state if any stage stops early.

### Recommended Next Step
- Use `/research-astack` to map the orchestration points that already exist in the repo and identify the smallest implementation surface for the unattended runner.

## Research

### Scope Baseline
- astack should support an unattended enhancement run that starts from scope and continues through ship, including docs sync, merge prompt, and local cleanup, while still stopping for anything that needs human judgment.

### Relevant Files
- `scope/SKILL.md.tmpl`
  - Planning entrypoint that already knows how to bootstrap enhancement branches.
- `research/SKILL.md.tmpl`
  - Discovery entrypoint that should reuse the same branch bootstrap and artifact handoff.
- `plan/SKILL.md.tmpl`
  - Decision-complete planning entrypoint that defines the implementation and QA matrix.
- `implement/SKILL.md.tmpl`
  - Execution entrypoint that updates progress and keeps the release artifact current.
- `qa/SKILL.md.tmpl`
  - Browser QA workflow that validates the enhancement before ship.
- `ship/SKILL.md.tmpl`
  - Release workflow that creates the PR, syncs docs, asks about merge, and performs cleanup.
- `docs/skills.md`, `README.md`, `CLAUDE.md`, `AGENTS.md`
  - User-facing workflow docs that need the unattended story to match the actual behavior.
- `test/gen-skill-docs.test.ts`, `test/skill-e2e.test.ts`, `test/skill-routing-e2e.test.ts`
  - Coverage points for the updated workflow language and the end-to-end journey.

### Current System Map
- The planning and execution skills already share a release-artifact model under `docs/releases/<version>-<slug>.md`.
- Scope, research, plan, and implement all assume the enhancement branch exists or should be bootstrapped when the workflow starts on `main`.
- QA is the dedicated verification stage, and ship is the release gate that already owns PR creation and post-ship cleanup.
- The repo already has tests that exercise skill generation, routing, and end-to-end skill prompts.

### Reuse Opportunities
- Reuse the active release artifact instead of introducing a second workflow state file.
- Reuse the existing branch bootstrap and ship cleanup behaviors instead of duplicating branch management.
- Reuse the current QA and ship skills as the automation checkpoints between stages.
- Reuse the skill generation pipeline so the `.tmpl` sources stay aligned with generated outputs.

### External References
- Git branch operations used by the existing workflow: `git switch`, `git branch -d`, `git pull --ff-only`.
- GitHub CLI behavior for PR creation, merge, and branch deletion.

### Constraints
- The unattended runner must not hide human-choice prompts.
- One retry is acceptable for transient errors, but not for branch conflicts or merge ambiguity.
- The implementation has to preserve the repo's current skill/branch conventions.

### Risks
- If orchestration is too eager, it could push through a failure that should have stopped the flow.
- If orchestration is too conservative, it may not feel meaningfully unattended.
- The release artifact needs to stay readable as the canonical handoff between stages.

### Assumptions To Validate
- The enhancement slug should be `end-to-end`.
- The unattended path should call the existing skills in order rather than reimplementing their internals.
- Retry-once behavior should only apply to clearly transient failures.

### Unknowns
- Whether the unattended orchestration belongs in a new skill surface, a release-artifact runner, or a documented internal workflow.
- Whether any branch bootstrap or merge cleanup behavior needs to be factored into shared helpers.

### Recommended Direction
- Keep the first version narrow: wire an unattended enhancement runner around the existing workflow stages, make retry handling conservative, and preserve all current safety gates.

## Plan

### Summary
- Build a new `end-to-end` enhancement around the existing astack workflow.
- The workflow should be able to take one enhancement request from scope through ship with minimal supervision, but it must stop on anything that needs a human decision.
- The implementation should reuse the existing skills as the actual stage executors and use the release artifact to pass state forward.

### What Already Exists
- `/scope-astack` through `/ship-astack` already define the individual workflow stages.
- `/scope-astack` and the downstream planning skills already bootstrap or reuse the enhancement branch when the workflow starts on `main`.
- `/ship-astack` already handles PR creation, docs sync, merge prompting, and cleanup after merge.
- The test suite already covers the generated skill docs and the routed skill journeys.

### Architecture
- Add an unattended orchestration path that sequences the existing skills in order:
  - scope
  - research
  - plan
  - implement
  - QA
  - ship
- Carry the active release artifact through the whole flow so each stage writes to the same enhancement record.
- Define a single retry policy for transient failures:
  - retry once when a command, test, or network step fails in a way that looks transient
  - do not retry when the failure is a human decision, branch conflict, or merge ambiguity
- Keep the orchestration layer thin so the stage-specific skills still own their own behavior and guardrails.

### Implementation Phases
#### Phase 1: add the unattended enhancement record
- Create a new versioned release artifact for `end-to-end`.
- Capture the problem, audience, goal, constraints, wedge, and risks for the unattended path.

#### Phase 2: document the workflow shape
- Update the workflow docs and skill descriptions so the unattended path is described consistently.
- Make clear that the automation still stops for any decision that needs the user.

#### Phase 3: validate the workflow surface
- Add tests that confirm the new artifact exists.
- Add tests that the generated skill docs still advertise branch bootstrap, merge prompting, and cleanup.
- Add end-to-end coverage for the new unattended story and retry boundary.

### QA And Test Matrix
- Scope path on `main` creates or reuses `enhancement/<slug>`.
- Research and plan continue to use the same enhancement branch and artifact.
- Implement updates progress without losing the artifact context.
- QA is invoked as the verification checkpoint before ship.
- Ship still prompts before merge and performs local cleanup after merge.
- Transient failure retries happen once and then stop if the problem persists.

### Out Of Scope
- New branch-management UI.
- Fully autonomous handling of ambiguous failures.
- Auto-merge without user confirmation.

### Open Questions
- Whether the unattended orchestration should be user-facing as a new skill or internal to existing workflow helpers.
- Whether any shared helper should be added to centralize retry decisions.
