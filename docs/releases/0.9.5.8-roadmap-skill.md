# Release

- Version: 0.9.5.8
- Status: shipped
- Last updated: 2026-03-27

## Scope

### Problem
- astack has strong scope, research, and plan skills, but no first-class workflow for turning a large request into a multi-release roadmap.
- Users who want phased planning have to force-fit `/plan-astack` or keep the release sequencing in chat, which makes the output harder to reuse and easier to lose.
- The roadmap skill also needs to exist in the user's global Codex skill install, not just in this repo's generated sidecars.

### Audience
- Primary user: an astack user who wants to plan work across multiple releases before picking one release to implement.
- Secondary stakeholders: astack maintainers who need the roadmap skill to be discoverable across Claude, Codex, and Copilot surfaces.

### Current Status Quo
- astack ships `/scope-astack`, `/research-astack`, `/plan-astack`, and `/implement-astack`, but not a roadmap-first alternative.
- The root skill and docs did not advertise a roadmap workflow.
- The global Codex install under `~/.codex/skills` did not contain `roadmap-astack`.
- This `0.9.5.8` draft was previously aimed at a PowerShell/Bun investigation and needed to be repurposed to match the actual enhancement work.

### Goal
- Add `roadmap-astack` as a first-class skill.
- Make the roadmap output a single document under `docs/roadmaps/<slug>.md` with `Summary Of Changes` first and then per-release sections inside the same file.
- Regenerate the Claude, Codex, and Copilot skill surfaces and refresh the user's global Codex install so `roadmap-astack` is available outside the repo.

### Constraints
- Preserve the existing `docs/releases/` contract for shipped enhancement artifacts.
- Keep roadmap planning separate from shipped release versioning by using `docs/roadmaps/`.
- Avoid disturbing unrelated user changes already present in the worktree.
- Keep the roadmap skill lighter than a full `/plan-astack` implementation spec.

### Narrow Wedge
- Add the roadmap skill template and generated outputs.
- Update discovery docs and test inventory so the skill is first-class everywhere astack lists its workflow surface.
- Refresh `~/.codex/skills` so the new skill is globally available.

### Non-Goals
- Reworking the existing scope/research/plan/implement workflows.
- Changing the roadmap output back to multiple files; the user explicitly wants one document with sections.
- Shipping or merging this enhancement in this turn.

### Risks And Unknowns
- The Windows sandbox on this machine blocks some `.agents` writes and bun-in-bun test spawning, so verification has to lean on direct command runs rather than every existing test path.
- The global Codex skill directory already contains older installs and legacy `astack-*` entries from previous setups.
- The branch name still reflects the older PowerShell investigation even though the artifact scope is now roadmap-skill work.

### Recommended Next Step
- Finish the roadmap-skill implementation, verify the generated outputs directly, and refresh the global Codex skill install from this repo.

## Research

### Scope Baseline
- This enhancement adds a roadmap-first planning skill, updates the generated host surfaces and docs that advertise skills, and makes sure the global Codex install exposes `roadmap-astack`.

### Relevant Files
- `roadmap/SKILL.md.tmpl`
  Why it matters: defines the new roadmap workflow and its single-document output contract.
- `scripts/gen-skill-docs.ts`
  Why it matters: generates the Claude, Codex, and Copilot skill surfaces and controls which skills Copilot receives.
- `SKILL.md.tmpl`
  Why it matters: advertises when the root astack skill should suggest the roadmap workflow.
- `README.md`
  Why it matters: documents the workflow surface and the roadmap skill's intended output.
- `AGENTS.md`
  Why it matters: repo-local instructions need to list the new skill and its output contract.
- `CLAUDE.md`
  Why it matters: contributor docs need to explain the roadmap output location.
- `docs/skills.md`
  Why it matters: the deep-dive workflow docs need to teach when to use the roadmap skill.
- `docs/host-support.md`
  Why it matters: the host inventory needs to include the new Copilot and Codex skill surface.
- `test/gen-skill-docs.test.ts`
  Why it matters: verifies skill inventory, root-skill suggestions, and generated host outputs.
- `test/skill-validation.test.ts`
  Why it matters: checks the generated skill health and Codex sidecar surface.
- `test/helpers/touchfiles.ts`
  Why it matters: routes E2E test selection when roadmap files change.
- `test/skill-routing-e2e.test.ts`
  Why it matters: adds routing coverage for `/roadmap-astack`.
- `setup`
  Why it matters: documents how Codex global installs are supposed to be refreshed from generated `.agents/skills/`.

### Current System Map
- astack source skills live in top-level folders and are generated from `SKILL.md.tmpl` files.
- `scripts/gen-skill-docs.ts` emits:
  - source-tree `SKILL.md` files for Claude-style use
  - `.agents/skills/<name>/SKILL.md` for Codex
  - `.copilot/skills/<name>/SKILL.md` for Copilot
- The global Codex install lives under `~/.codex/skills`:
  - `astack/` acts as the runtime root
  - each generated skill is exposed as its own directory like `plan-astack/`
- The install state before this change still pointed the runtime metadata back to an older checkout at `/c/Users/jcteb/src/astack`.

### Reuse Opportunities
- The roadmap skill can reuse the existing astack preamble and planning-style structure instead of adding new scripts.
- The host generator already knows how to emit Codex and Copilot versions of a new skill once the source template exists.
- The current global Codex install can be refreshed by copying generated skill docs and runtime metadata rather than inventing a separate install path.

### External References
- None required beyond the repo's current setup and host-support docs; this enhancement is driven by astack's internal workflow model.

### Constraints
- Normal shell writes into `.agents/skills/` were blocked by the Windows sandbox in this session, so Codex regeneration required escalated execution.
- The existing `bun test` dry-run freshness tests hit an environment-specific Bun spawn error on this machine, so direct dry-run commands are the reliable freshness check here.
- The global skill directory is outside the repo, so refreshing it requires escalated writes under the user's home directory.

### Risks
- Docs and generated host outputs can drift if the roadmap skill is added only in source and not regenerated everywhere.
- The global skill directory can still expose stale behavior if only repo-local sidecars are updated.
- Legacy global `astack-*` directories from an older install model may remain alongside the canonical `*-astack` skill names.

### Assumptions To Validate
- The user wants the roadmap output as one document with sections, not a multi-file roadmap package.
- The global target that matters for this enhancement is Codex under `~/.codex/skills`.
- Direct dry-run generator commands are sufficient verification for freshness in this session.

### Unknowns
- Whether the legacy global `astack-*` entries should be cleaned up in a separate follow-up.
- Whether the branch itself should be renamed later to match the retargeted 0.9.5.8 scope.

### Recommended Direction
- Keep the roadmap contract as one document under `docs/roadmaps/<slug>.md`.
- Treat the generated `.agents` and `.copilot` surfaces as required ship artifacts for this skill.
- Refresh the global Codex install directly from this repo after regeneration so the new skill is immediately usable.
