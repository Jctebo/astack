## Scope

### Problem
- Starting a new enhancement still requires too much manual branch handling.
- Users can begin work without a clean branch created from `main`, which makes it easier to drift away from the baseline they meant to build from.
- Shipping also stops short of the final landing step. A PR can be created, but the user still has to remember to decide whether to merge it, clean up the branch, and return to `main`.
- The result is a workflow that can leave people on the wrong branch or leave old feature branches behind after the work is finished.

### Audience
- Primary user: astack maintainers and contributors who repeatedly start, ship, and land enhancements.
- Secondary stakeholders: reviewers and teammates who benefit from predictable PR creation, branch cleanup, and a workspace that returns to `main` after merge.

### Current Status Quo
- New enhancement work does not automatically guarantee a fresh branch from `main`.
- `/ship-astack` already creates a PR, but the loop ends before merge decision, branch deletion, and return-to-main cleanup are handled.
- Branch cleanup is manual, so feature branches can accumulate after shipping.

### Goal
- Automatically start each new enhancement on a fresh branch created from `main`.
- On ship, create the PR and then ask the user whether they want to merge it.
- After a merge, delete the feature branch and switch the workspace back to `main`.
- Keep the workflow obvious and safe so the user always knows what branch they are on.

### Constraints
- Must preserve uncommitted work safely when switching or creating branches.
- Must not delete the feature branch until after the user has had a chance to merge or explicitly decline.
- Must remain compatible with the existing release-artifact workflow and current ship/review/plan flow.
- Must handle cases where a branch or PR already exists for the enhancement.

### Narrow Wedge
- Automate branch creation only at the start of a new enhancement, not across every workflow command.
- Make `/ship-astack` the place where the PR is opened, the merge question is asked, and branch cleanup is triggered.
- Optimize the first version for the common path: `main` -> feature branch -> PR -> optional merge -> delete branch -> return to `main`.

### Non-Goals
- Replacing GitHub PRs with another hosting provider.
- Building a branch-management UI.
- Auto-merging without explicit user confirmation.
- Supporting complex multi-branch release trains or long-lived parallel release branches.

### Risks And Unknowns
- Automatic branch creation could be awkward if the workspace is dirty or if the user already has local changes in progress.
- Deleting branches too aggressively could remove work the user still wants if the merge decision or branch target is wrong.
- The best hook point for "starting a new enhancement" still needs to be pinned down in the workflow.
- We need to confirm whether branch creation belongs in `/scope-astack`, `/plan-astack`, or a separate start-of-work trigger.

### Recommended Next Step
- Use `/research-astack` to map where enhancement starts are detected today, how `/ship-astack` currently creates PRs, and which branch switch/delete operations are safe to automate.

## Research

### Scope Baseline
- astack should automatically create a fresh feature branch from `main` when a new enhancement starts, then let `/ship-astack` create the PR, ask whether to merge it, delete the branch after merge, and return the workspace to `main`.

### Current System Map
- The planning skills already share a release-artifact workflow and know how to resolve the active `docs/releases/<version>-<slug>.md` file.
- None of the planning skills currently create or switch branches; they assume the current branch is already the right place to write the enhancement record.
- `/ship-astack` already:
  - detects the base branch
  - runs checks and review gates
  - pushes the feature branch
  - creates a PR
  - auto-invokes `/document-release-astack`
  - stops before merge, branch deletion, or return-to-main cleanup
- `/document-release-astack` already assumes it runs before PR merge, so the current doc-sync sequencing can stay intact if merge confirmation happens after docs are synced.
- `setup` and `scripts/gen-skill-docs.ts` are not branch-aware today; they only build and distribute the skill content.
- The existing test suite already exercises skill generation, routing, and several end-to-end skill flows, which gives us a natural place to add branch-lifecycle coverage.

### Reuse Opportunities
- Reuse the active release artifact as the source of truth for branch naming, so branch creation can key off the enhancement slug that already exists.
- Reuse the existing base-branch detection logic in `/ship-astack` and `/document-release-astack` rather than inventing a second notion of the repo default branch.
- Reuse the existing `gh pr create`/`gh pr merge` workflow rather than adding a custom branch-management command.
- Reuse the generated skill pipeline so the source `.tmpl` edits fan out to Claude, Codex, and Copilot outputs in one pass.

### External References
- GitHub CLI behavior for `gh pr create`, `gh pr merge`, and `--delete-branch`.
- Git branch commands for `git switch -c`, `git switch`, `git branch -d`, and `git pull --ff-only`.

### Constraints
- The branch bootstrap must not silently discard dirty worktrees.
- The ship flow already has a pre-merge documentation step, so the merge prompt needs to come after docs are synced and pushed or the PR body will be stale.
- Deleting the branch only makes sense after a successful merge; declining merge should leave the branch and PR open.
- The implementation has to keep all generated host skill outputs in sync with the source templates.
- The branch naming scheme needs to be deterministic from the release artifact so repeated runs can reuse the same enhancement branch.

### Risks
- Highest risk: branch cleanup runs against the wrong branch or before the merge is actually complete.
- High risk: switching back to `main` after remote merge leaves the local workspace out of sync if the branch deletion or fast-forward step is incomplete.
- Medium risk: adding merge confirmation to `/ship-astack` changes a flow that is currently fully automated, so the UX wording has to be crisp and unambiguous.
- Medium risk: branch bootstrap in multiple planning entrypoints can drift if the behavior is copied manually into several templates.

### Assumptions To Validate
- The enhancement branch should be named `enhancement/<slug>` and reused if it already exists.
- The first workflow touchpoint on `main` should auto-create the branch; later stages should reuse it.
- A dirty worktree should stop the automatic branch switch rather than trying to stash and restore changes.
- The merge prompt should offer a simple yes/no choice, with "yes" using the repository's normal merge path.

### Unknowns
- Whether `gh pr merge --merge --delete-branch` should be followed by a local `git branch -d` or whether the remote deletion plus `git switch main` is enough for this repo's workflow.
- Whether any downstream docs besides the workflow docs need explicit wording changes for the new merge-confirmation step.

### Recommended Direction
- Add a small branch-bootstrap guard to `scope`, `research`, `plan`, and `implement` so the first touch on `main` creates `enhancement/<slug>`, then extend `ship` so PR creation is followed by docs sync, a merge confirmation prompt, and post-merge cleanup back to `main`.

## Plan

### Summary
- Build a branch-lifecycle workflow on top of the existing release-artifact flow.
- The first planning touch on `main` should auto-create or reuse a deterministic feature branch for the active enhancement.
- `/ship-astack` should keep doing the PR and doc-sync work it already owns, then ask whether to merge, and if the user says yes it should merge, delete the branch, and return the workspace to `main`.
- This is the chosen approach because it keeps branch management inside the same workflow the user already runs, instead of asking them to remember a separate branch command.

### What Already Exists
- The release artifact contract already tracks the active enhancement in `docs/releases/<version>-<slug>.md`.
- `/ship-astack` already knows how to detect the base branch, push the feature branch, create the PR, and run `/document-release-astack`.
- `/document-release-astack` already expects to run before merge and can continue to do so.
- The generated skill pipeline already keeps `.tmpl` sources and host-specific `SKILL.md` outputs in sync.
- The test suite already has coverage for generated skill docs, workflow routing, and end-to-end skill flows.

### Architecture
- Add a shared branch-bootstrap step to the planning entrypoints:
  - read the active release artifact to get the enhancement slug
  - if the current branch is `main`, create `enhancement/<slug>` from `main`
  - if the branch already exists, switch to it instead of creating a duplicate
  - if the worktree is dirty, stop and tell the user to resolve the local changes first
- Keep the merge decision inside `/ship-astack`:
  - PR creation and docs sync happen first
  - then ask whether to merge now
  - on yes, merge with the repo's normal merge method, delete the branch, fast-forward local `main`, and switch back to it
  - on no, leave the PR open and keep the feature branch checked out
- Treat the branch name as derived data from the release slug so the same enhancement always maps to the same branch.

```text
main
  |
  | start scope/research/plan/implement
  v
enhancement/<slug>
  |
  | build + review + qa
  v
/ship-astack
  |
  | create PR
  | run document-release
  v
merge now?
  | yes                                  | no
  v                                      v
gh pr merge --merge --delete-branch     keep PR open
  |
  v
git switch main -> git pull --ff-only -> delete local branch
```

### UX And States
- User starts on `main` and invokes `/scope-astack`:
  - if the tree is clean, astack switches to `enhancement/<slug>` and continues
  - if the tree is dirty, astack stops and explains that it cannot safely create the branch yet
- User continues through research, plan, and implement:
  - astack reuses the same enhancement branch
  - it does not create a second branch for the same enhancement
- User runs `/ship-astack`:
  - PR is created
  - docs sync runs before the merge prompt so the PR includes the final release docs
  - user is asked whether to merge now
  - if yes, the PR is merged, the branch is deleted, and astack returns to `main`
  - if no, the PR stays open and the branch remains checked out

### Failure Modes
- The worktree is dirty on `main`, so auto-branching stops and asks the user to resolve local changes first.
- The enhancement branch already exists but points at the wrong work; reuse needs to be explicit so the user does not accidentally fork the enhancement.
- The PR merge step fails or is declined; in that case the branch should stay open and the workflow should not pretend cleanup succeeded.
- The remote merge succeeds but local `main` cannot fast-forward cleanly; the workflow should stop rather than guessing.
- Branch cleanup is attempted before the merge is complete; the workflow should treat that as a hard failure.

- `scope` on `main`

- `research`, `plan`, `implement` on `main`

- `ship` on a feature branch with a clean tree

- existing-branch case

- regression checks
  - generated Claude/Codex/Copilot skill docs contain the new branch-lifecycle wording and no unresolved placeholders
  - branch lifecycle instructions are present in the workflow docs

### Rollout Notes
- No flag or migration is needed; the workflow changes land with the regenerated skill docs.
- The new behavior should ship as a single coordinated pass so the source templates, generated outputs, docs, and tests all describe the same branch lifecycle.
- Because `/ship-astack` becomes interactive at the merge decision, the PR body and docs-sync step should remain ahead of the prompt so the PR reflects the final state.

### Not In Scope
- Auto-merging without asking the user.
- Custom branch naming preferences or a branch-name settings UI.
- Supporting multiple concurrent enhancement branches in one workspace.
- Reworking GitHub PR creation into a different hosting flow.

### Open Questions
- None. The branch name, merge method, and cleanup order are now pinned down enough for implementation.
