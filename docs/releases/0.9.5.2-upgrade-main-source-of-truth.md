## Scope

### Problem
- `/astack-upgrade-astack` is not reliably upgrading the primary astack install on the user's machine.
- In Codex, the upgrade workflow can resolve `.agents/skills/astack` or other repo-local copies instead of the real host install under `~/.codex/skills/astack`.
- The result is that users can run upgrade in a fresh window and still end up with stale or mismatched astack behavior because the wrong directory was treated as the source of truth.
- This matters now because the upgrade path is the trust anchor for keeping all generated skills and runtime helpers in sync with `main`.

### Audience
- Primary user: astack users on Codex who expect `/astack-upgrade-astack` to update the real install they are invoking.
- Secondary stakeholders: astack maintainers who need one unambiguous install model across setup, upgrade, docs, and generated host outputs.

### Current Status Quo
- Codex install docs say the main install root is `~/.codex/skills/astack`.
- Before this change, `setup` created a slim runtime root at `~/.codex/skills/astack` and repo-local `.agents/skills/astack` sidecars for generated skills without recording which source checkout actually owned upgrades.
- Before this change, the generated Codex upgrade skill still contained install-detection logic that prioritized `.agents/skills/astack`-style paths as primary install candidates.
- Before this change, the upgrade flow also assumed the chosen install directory could be reset to `origin/main` and re-run with `./setup`, which made runtime roots and repo-local sidecars unsafe upgrade targets.

### Goal
- Make upgrade behavior target the real astack source-of-truth install for the current host, not whichever local sidecar happens to be present.
- Ensure Codex upgrades are anchored to `main` in the intended primary install location.
- Preserve the ability to sync vendored copies after the primary install has been upgraded, but only as a secondary step.

### Constraints
- The install model differs by host: Claude, Codex, and Copilot do not share the same runtime layout.
- Codex intentionally uses a slim runtime root plus generated sidecar skill folders, so upgrade logic cannot assume every discovered astack directory is a full git checkout.
- The upgrade flow touches user-home installs and repo-local copies, so mistakes are high-impact.
- Setup and upgrade docs must stay aligned with generated host-specific skill output.

### Narrow Wedge
- Fix the Codex upgrade path first.
- Define one primary install target for Codex upgrades.
- Make install detection explicitly distinguish:
- the primary host install
- a repo-local vendored copy
- a runtime root or generated sidecar that should not be treated as the upgrade source
- Keep vendored-copy sync as an opt-in secondary action after the primary install is upgraded from `main`.

### Non-Goals
- Redesigning every host's install UX in this pass.
- Changing the `astack-update-check` version-check protocol itself.
- Solving all Windows setup issues unrelated to install-target detection.
- Reworking unrelated workflow skills outside upgrade/setup/install discovery.

### Risks And Unknowns
- The current Codex install docs and `setup` behavior may already encode a deeper mismatch between the documented source checkout and the runtime root they generate.
- If the real primary install location is ambiguous today, a narrow fix in the upgrade skill alone may not be enough.
- There is a risk of destructive behavior if upgrade keeps using `git stash` and `git reset --hard origin/main` against the wrong directory type.

### Recommended Next Step
- `/research-astack` mapped the exact relationship between Codex install docs, `setup`, generated upgrade skills, and runtime roots, which led directly into the implemented contract below.

### Scope Baseline
- This research covers why `/astack-upgrade-astack` in Codex can resolve repo-local astack directories instead of the intended primary install and how to anchor upgrades to `main`.

### Current System Map
- `README.md` previously told Codex users to clone astack into `~/.codex/skills/astack` and run `./setup --host codex`.
- `docs/host-support.md` repeated that Codex install root is `~/.codex/skills/astack`, while `.agents/skills/` was described as generated source in the repo rather than the canonical host install.
- Before implementation, `setup` installed Codex in two layers:
  - a slim runtime root at `~/.codex/skills/astack` containing the root `SKILL.md` plus symlinked `bin` and `browse/dist`
  - generated Codex skill folders symlinked into `~/.codex/skills/<skill-name>`
- `setup` also created a repo-local `.agents/skills/astack` runtime root for workspace-local use, built from generated Codex output rather than the full source checkout.
- This means there are three different Codex-adjacent astack path shapes in play:
  - source checkout used to run `setup`
  - user-home runtime root at `~/.codex/skills/astack`
  - repo-local runtime sidecar at `.agents/skills/astack`
- Before implementation, the generated Codex upgrade skill detected install type using `.agents/skills/astack` paths as if they were primary install candidates, including `$HOME/.agents/skills/astack` and local `.agents/skills/astack`.
- Before implementation, the generated Codex upgrade skill then treated the detected directory as a git checkout or full install that could be upgraded with `git stash`, `git fetch`, `git reset --hard origin/main`, and `./setup`.
- `bin/astack-update-check` expects an actual astack root with `VERSION`, `bin/astack-config`, and access to remote `main` version metadata.

### Reuse Opportunities
- Reuse `docs/host-support.md` as the contract source for host install roots rather than inventing new upgrade-only path rules.
- Reuse `setup`'s explicit host separation and runtime-root concept instead of treating every astack-looking directory as equivalent.
- Reuse the existing version-check state files (`just-upgraded-from`, `last-update-check`, `update-snoozed`) once the primary install target is unambiguous.

### External References
- `README.md` documents Codex install as `~/.codex/skills/astack`.
- `docs/host-support.md` documents Codex install root as `~/.codex/skills/astack` and `.agents/skills/` as generated source in the repo.

### Constraints
- Facts: Codex host paths in `scripts/gen-skill-docs.ts` use `~/.codex/skills/astack` for `skillRoot` and `.agents/skills/astack` for `localSkillRoot`.
- Facts: `.agents/skills/astack` in a repo is a runtime sidecar, not necessarily the full git checkout the user expects to upgrade.
- Facts: the generated Codex upgrade skill currently searches `.agents/skills/astack` paths in Step 2 and does not search `~/.codex/skills/astack` at all.
- Facts: `setup` creates a runtime root at `~/.codex/skills/astack` by copying generated root skill content and symlinking selected runtime assets, not by preserving the full source checkout as the upgrade contract.
- Inference: this mismatch is likely why a new Codex window can believe astack is installed locally to a repo and then operate on the wrong directory.

### Risks
- Highest risk: destructive upgrade operations (`git stash`, `git reset --hard origin/main`) run against the wrong path.
- High risk: setup and upgrade continue to disagree about whether `~/.codex/skills/astack` is a source checkout, a runtime root, or both.
- High risk: fixing only the generated Codex upgrade skill could leave README/setup semantics inconsistent and recreate the issue later.

### Assumptions To Validate
- The intended primary upgrade target for Codex should be the actual source checkout that tracks `main`, not a generated runtime root or repo-local `.agents/skills/astack` copy.
- `~/.codex/skills/astack` is currently described like a source checkout in docs, but `setup` treats it like a runtime root destination.
- Vendored-copy sync should happen only after the primary install is upgraded successfully.

### Unknowns
- Whether the current Codex setup flow is safe when the source checkout itself lives at `~/.codex/skills/astack`.
- Whether the right long-term model is:
- keep a dedicated source checkout separate from the runtime root, or
- keep `~/.codex/skills/astack` as the source checkout and install runtime assets elsewhere
- Whether similar misdetection exists for Claude or Copilot once their generated upgrade skills are exercised.

### Recommended Direction
- Treat this as an install-contract bug, not just a stale-upgrade-script bug.
- The shipped change defined the Codex upgrade source of truth as the source checkout recorded by the user-home runtime root.
- The shipped contract now treats:
  - `~/.codex/skills/astack` as the Codex runtime entrypoint
  - `.astack-source-path` inside that runtime root as the authoritative pointer to the real source checkout
  - `.agents/skills/astack` as a derived repo-local runtime or vendored copy, not a primary upgrade source
- The shipped upgrade flow now:
  - positively identifies the user-home Codex runtime root first
  - upgrades the recorded source checkout on `main`
  - refuses to treat repo-local runtime roots as primary installs
  - optionally syncs vendored copies afterward
- README, host-support docs, setup, and generated upgrade skill text were updated together so the install model now aligns.

## Plan

### Summary
- Make Codex upgrades operate against one explicit source of truth instead of inferring from repo-local `.agents/skills/astack` paths.
- Use the Codex user-home install root as the upgrade entrypoint, but resolve and upgrade the actual source checkout that tracks `main`.
- Preserve support for legacy installs where the source checkout itself still lives at `~/.codex/skills/astack`.

### What Already Exists
- `setup` already knows how to install Codex runtime artifacts into `~/.codex/skills/`.
- `scripts/gen-skill-docs.ts` already centralizes Codex path generation for generated skills.
- `bin/astack-update-check` already supports a clear local-root contract once it points at a directory with `VERSION` and `bin/astack-config`.
- The release-doc workflow is already available for documenting the install contract change.

### Architecture
- Introduce an explicit Codex source-checkout contract:
  - Codex runtime root: `~/.codex/skills/astack`
  - Codex source checkout: a git checkout that tracks `main`
- Persist the source checkout path into the Codex runtime root so upgrades can find the real git repo instead of guessing.
- Treat `.agents/skills/astack` only as a derived repo-local runtime artifact or vendored copy.
- Keep legacy compatibility:
  - if `~/.codex/skills/astack` itself is a git checkout, treat it as a legacy source install and upgrade it directly
  - otherwise prefer the persisted source path metadata

### Concrete Changes
- Update `setup` to write Codex source-path metadata into `~/.codex/skills/astack`.
- Update `setup` so the Codex runtime root also exposes enough metadata for version/readme-style tooling to identify the installed version cleanly.
- Update `astack-upgrade/SKILL.md.tmpl` to:
  - detect `~/.codex/skills/astack` as the Codex primary install entrypoint
  - resolve the real source checkout from metadata or legacy in-place git state
  - stop treating `.agents/skills/astack` as a primary install candidate
  - keep `.agents/skills/astack` only as an optional post-upgrade sync target
- Regenerate generated upgrade skills and any host-facing docs affected by that template change.
- Update `README.md` and `docs/host-support.md` to describe the Codex install/upgrade source of truth clearly.
- Add or update tests around upgrade/install detection and setup semantics where practical.

### UX And States
- Direct Codex install:
  - user upgrades astack
  - upgrade resolves the real Codex source checkout
  - upgrade fast-forwards that checkout to `origin/main`
  - setup refreshes runtime artifacts under `~/.codex/skills/`
- Legacy in-place install:
  - if `~/.codex/skills/astack` is itself a git checkout, upgrade uses it directly
- Repo-local vendored copy present:
  - upgrade completes on the primary source checkout first
  - vendored `.agents/skills/astack` sync happens only as a secondary step
- Missing or invalid source metadata:
  - upgrade should fail clearly instead of silently falling back to a repo-local sidecar

### Failure Modes
- Upgrade still resolves `.agents/skills/astack` as primary.
  - Detection: generated upgrade skill still mentions `.agents/skills/astack` in primary detection.
  - User impact: upgrades operate on repo-local copies instead of the real install.
- Codex runtime root lacks valid source metadata.
  - Detection: upgrade cannot resolve a git checkout from the runtime root.
  - User impact: update prompts continue, but upgrades fail or noop.
- Legacy in-place installs break.
  - Detection: upgrade fails when `~/.codex/skills/astack/.git` exists.
  - User impact: older installations cannot upgrade without manual repair.
- Docs and setup diverge again.
  - Detection: README/setup/host-support describe different install roots.
  - User impact: users follow one path while upgrade assumes another.
