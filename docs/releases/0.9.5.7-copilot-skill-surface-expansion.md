# Release

- Version: 0.9.5.7
- Status: shipped
- Last updated: 2026-03-24

## Scope

### Problem
- GitHub Copilot currently gets only the base planning workflow plus `/end-to-end`, while the repo has a much larger astack skill surface for Claude and Codex.
- That makes Copilot the weakest host for users who want the same guided workflow they see elsewhere in astack.
- The product question is not “should Copilot get everything forever,” but “what additional astack skills should Copilot ship now so the host feels meaningfully useful without overpromising parity?”

### Audience
- Primary user: a GitHub Copilot user working inside a repo that has astack installed and wants more than just scope/research/plan/implement.
- Secondary stakeholders: astack maintainers who need a stable host boundary, generated docs, and tests that prevent drift.

### Current Status Quo
- Copilot installs only `/scope-astack`, `/research-astack`, `/plan-astack`, `/implement-astack`, and `/end-to-end`.
- Operational workflows like review, QA, ship, docs sync, design, investigate, and upgrade remain unavailable on that host.
- The generated docs and tests already encode this smaller boundary, so the gap is intentional rather than accidental.
- The Copilot support surface is now being expanded to the full astack skill set, so the old “planning-only” contract is no longer the target.

### Goal
- Put the full astack skill surface on Copilot so the host matches the rest of the repo’s workflow vocabulary.
- Preserve a clear host contract so the Copilot surface is still explainable and testable.
- Make the resulting capability visible in generated docs, host-support docs, and validation tests.

### Constraints
- Reuse the existing skill generation pipeline instead of creating a separate Copilot-specific install path.
- Keep canonical skill names and generated output conventions intact.
- Keep the generated Copilot set deterministic and in sync with the source templates.

### Narrow Wedge
- Expand Copilot in one deliberate step by shipping the full astack skill set to the host.
- Treat `/end-to-end` as part of the now-complete Copilot workflow surface rather than the only operational exception.
- Prefer a single coherent host policy over piecemeal additions.

### Non-Goals
- Introducing a new installation mechanism.
- Rewriting the skill generation system or inventing a new installation model.
- Changing the meaning of existing planning skills.

### Risks And Unknowns
- Whether the larger Copilot surface will overwhelm users who only wanted planning support.
- How much documentation and test coverage needs to move together if the host boundary changes.

### Recommended Next Step
- Use `/research-astack` to map the Copilot generation path, the current host boundary docs/tests, and the full astack skill inventory that should be generated for Copilot.

## Research

### Scope Baseline
- Copilot should ship the full astack surface, not a reduced planning-only bundle.
- This enhancement is about aligning the Copilot host surface with the repo’s generated skill inventory, not rewriting the workflow model.
- The current boundary is explicit, not emergent: `scripts/gen-skill-docs.ts` filters Copilot output through a fixed allowlist, and the test suite asserts the exact Copilot skill set.

### Relevant Files
- `docs/host-support.md` - defines the current Copilot support boundary and the host contract.
- `README.md` - advertises the current Copilot install surface to users.
- `docs/skills.md` - documents the workflow split and host-specific behavior.
- `scripts/gen-skill-docs.ts` - generates `.copilot/skills/` output and encodes which skills Copilot receives.
- `test/gen-skill-docs.test.ts` - asserts the Copilot output set and freshness rules.
- `test/skill-validation.test.ts` - checks that generated skill docs and host support stay aligned.
- `test/skill-routing-e2e.test.ts` - verifies workflow routing expectations that may change if the Copilot surface expands.
- `.copilot/skills/` - the generated host output that will need to match any new Copilot boundary.

### Current System Map
- Copilot host generation is driven by `scripts/gen-skill-docs.ts`, which filters the output set for the `copilot` host.
- The Copilot bundle should include every shipped astack skill directory except the self-referential `codex` host wrapper.
- Host boundaries are documented in `docs/host-support.md`, and the same boundary is echoed in `README.md` and `docs/skills.md`.
- Tests already assert the current Copilot skill list, so any expansion will require code, docs, and test updates together.
- `test/gen-skill-docs.test.ts` currently expects the Copilot set to be exactly the generated full surface, so the assertions need to move with the generator.
- `docs/host-support.md` should now describe Copilot as a full astack host instead of a smaller planning-only surface.

### Reuse Opportunities
- The existing skill-generation pipeline already supports host-specific output, so the change can likely be expressed as a filter or allowlist adjustment rather than a new mechanism.
- The existing `docs/host-support.md` format can be extended instead of replaced.
- The current test suite gives a ready-made place to encode the new Copilot boundary and prevent regressions.
- The operational skills already exist in the repo and in generated Codex/Claude surfaces, so the main reuse opportunity is policy selection, not new skill authoring.

### External References
- GitHub Copilot skill support model as documented by astack itself in `docs/host-support.md` and `README.md`.
- No external API or third-party library appears to be required for the first research pass.

### Constraints
- Copilot support is currently treated as intentionally narrower than Claude and Codex.
- Generated host outputs must stay in sync with the source templates and validation tests.
- Any expansion needs to preserve the canonical skill naming scheme used across hosts.

### Risks
- The larger Copilot surface could be noisier for users who only want the planning path.
- The more skills Copilot gets, the more likely the docs and tests become stale if the change is not made holistically.

### Assumptions To Validate
- Users actually want the full astack skill surface on Copilot, not just the existing planning flow.
- The generated Copilot surface can be broadened without requiring a separate installation model.

### Unknowns
- None remain for the host policy decision; the full skill set is now the target.

### Recommended Direction
- Update the generator, host docs, and tests together so Copilot ships the full astack skill surface without drift.

## Plan

### Summary
- Expand Copilot to the full astack skill surface so the host can use the same workflow vocabulary as the rest of the repo.
- The chosen surface is every shipped astack skill directory except the self-referential `codex` wrapper.
- This makes Copilot materially more useful by removing the host split altogether.

### What Already Exists
- `scripts/gen-skill-docs.ts` already generates host-specific output and can be expanded by adjusting the Copilot allowlist.
- `docs/host-support.md`, `README.md`, and `docs/skills.md` already describe the current Copilot boundary and can be updated together.
- The operational skills already exist in the repo for other hosts, so the work is mostly about host policy and docs/test synchronization.
- `test/gen-skill-docs.test.ts`, `test/skill-validation.test.ts`, and `test/skill-routing-e2e.test.ts` already provide enforcement points for the new boundary.

### Architecture
- Copilot generation remains a single pipeline:

```text
source SKILL.md.tmpl files
        |
        v
scripts/gen-skill-docs.ts
        |
        +--> .agents/skills/ (Codex)
        |
        +--> .copilot/skills/ (Copilot allowlist)
        |
        v
docs + tests assert the generated host surface
```

- The implementation changes the Copilot host policy, not the underlying skill model.
- The host docs become the user-facing contract, while the tests become the enforcement layer.

### Concrete Changes
- `scripts/gen-skill-docs.ts`
  - Expand `COPILOT_WORKFLOW_DIRS` to include every shipped astack skill directory except `codex`.
  - Keep the root `astack` skill and all workflow/support skills in the Copilot bundle.
- `docs/host-support.md`
  - Replace the current “planning workflow plus `/end-to-end`” boundary with the new Copilot bundle description.
  - Clarify which skills are now installed and which remain intentionally absent.
- `README.md`
  - Update the Copilot install section and the workflow list so the user-facing docs match the new host boundary.
- `docs/skills.md`
  - Update the host behavior section and any installation guidance to match the expanded Copilot surface.
- `test/gen-skill-docs.test.ts`
  - Update the expected Copilot skill list and freshness assertions.
- `test/skill-validation.test.ts`
  - Update the Copilot sidecar expectations and any host contract assertions that assume the smaller bundle.
- `test/skill-routing-e2e.test.ts`
  - Add or adjust routing coverage if the larger Copilot surface changes which skill should be suggested for review/QA/ship/document-release prompts.
- Generated output under `.copilot/skills/`
  - Regenerate host sidecars and ensure stale directories are cleaned up.

### UX And States
- User journey:
  - A Copilot user scopes and researches a problem, plans it, implements it, then can continue into review, QA, ship, and release docs without switching hosts.
- Empty state:
  - No special UI state; the Copilot host surface is a skill menu and generated docs contract.
- Success state:
  - The Copilot host exposes the expanded workflow in generated docs and the listed skills are discoverable.
- Error state:
  - If generation is stale or tests disagree with the allowlist, the repo fails validation instead of silently shipping a broken host contract.

### Failure Modes
- Copilot output still only contains the old planning set, which would mean the allowlist change did not take effect.
- The docs claim a broader Copilot surface but the generated `.copilot/skills/` tree or tests still reflect the smaller set.
- Routing tests may still steer operational prompts to other skills if the broader surface is not reflected in the suggestion logic.
- Users may see a broader host boundary in docs but not in install output if regeneration is skipped.

### QA And Test Matrix
- Verify generated Copilot skill directories exist for `astack` and every shipped skill directory except `codex`.
- Verify `bun run gen:skill-docs --host copilot --dry-run` reports fresh output.
- Verify `docs/host-support.md`, `README.md`, and `docs/skills.md` describe the same Copilot boundary.
- Verify test coverage for the expanded Copilot list passes in `test/gen-skill-docs.test.ts` and `test/skill-validation.test.ts`.
- Verify routing for prompts that mention review, QA, shipping, docs sync, design, debugging, and upgrades still maps to the intended skills in `test/skill-routing-e2e.test.ts`.
- Edge cases:
  - stale generated sidecars are cleaned up
  - no Claude paths leak into Copilot files
  - the Copilot list remains deterministic

### Rollout Notes
- No feature flag is planned; the change is a repo-level host policy update.
- The Copilot surface should be regenerated and tested in the same change so docs and outputs cannot drift.
- If the host surface needs to be narrowed later, the smallest rollback is to remove selected directories from `COPILOT_WORKFLOW_DIRS`.

### Not In Scope
- Rewriting the install system or introducing a new host type.
- Changing the behavior of the skills themselves.

### Open Questions
- None remain for the host surface decision.
