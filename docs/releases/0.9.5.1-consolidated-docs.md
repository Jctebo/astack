## Scope

### Problem
- astack currently spreads planning and execution context across separate repo-root artifacts like `00-scope.md`, `01-research.md`, `02-plan.md`, and `03-progress.md`.
- That model makes the workflow harder to follow inside a target repo because the artifacts are fragmented, easy to lose, and not grouped as one release or enhancement record.
- astack also lacks a clear release-folder contract for versioning, summarized history, and a single durable artifact that every skill can keep appending to.

### Why Now
- The current contract is still small enough to migrate before more skills and docs harden around the repo-root artifact pattern.
- A consolidated release model could make astack easier to adopt in target repos by giving teams one predictable place to inspect the current enhancement record.

### Audience
- Primary user: maintainers and contributors evolving astack's artifact contract.
- Secondary users: teams using astack inside a target repo who want one obvious documentation trail per enhancement.
- Secondary stakeholders: future skill authors who need a stable contract for where to read and write lifecycle artifacts.

### Current Status Quo
- Core astack planning assumes four numbered repo-root files that are written in sequence.
- Supporting docs, generated skills, and tests all reinforce that repo-root contract.
- Release notes in this repo are not yet the source of truth for active enhancement artifacts.

### Goal
- Define a new documentation contract centered on `docs/releases/`.
- Establish three durable release-folder artifacts:
- A version tracker that increments within the release folder.
- A summarized release log that captures the high-level history.
- One consolidated per-enhancement file, prefixed with the version number, that becomes the append target for astack-created artifacts.
- Make "which consolidated file are we updating?" an explicit confirmation step for skills working against this model.
- Make this the real versioning and long-term archival model for astack itself, not just a pattern for downstream target repos.
- Ensure shipped enhancement documentation becomes a past-tense historical record after ship and documentation sync workflows run.

### Success Criteria
- The new contract is simple enough to explain in one short section of docs.
- A contributor can identify the active enhancement artifact by looking only in `docs/releases/`.
- The lifecycle of scope, research, plan, implementation progress, QA, and release notes can be understood from one consolidated enhancement file.
- The version prefix is consistent enough that artifact ordering is obvious without extra tooling.
- Skills have a defined expectation to confirm the target consolidated file before appending content.
- After shipping, the consolidated artifact reads like an archive of what happened, not a draft of what might happen.

### Constraints
- This scope should define the contract before code or generator changes begin.
- The first iteration should stay documentation-first and avoid introducing a large migration framework.
- The release-folder model needs to work across astack skills, generated host outputs, and tests.
- Any new interaction model should keep skill prompts lightweight rather than adding repeated high-friction setup.

### Narrow Wedge
- Introduce a `docs/releases/` contract for astack-managed artifacts.
- Treat the release folder as the canonical home for:
- A monotonically incremented version record.
- A concise summary release log.
- A single version-prefixed consolidated file for each enhancement.
- Define that consolidated file as the place where all relevant astack skills append their durable output instead of creating multiple separate lifecycle files.
- Require skills to ask for or confirm the intended consolidated file before writing to it.
- Require `/ship-astack` and `/document-release-astack` to convert the consolidated enhancement document from in-flight planning language to shipped, past-tense release documentation.

### Non-Goals
- Implementing the full migration in this scope step.
- Designing the final prompt wording for every skill.
- Solving archival strategy for every past enhancement immediately.
- Reworking unrelated docs or release automation outside the artifact contract.

### Risks And Unknowns
- The current repo-root artifact contract is deeply embedded across skills, docs, and tests, so migration surface area may be larger than it first appears.
- Requiring every skill to confirm the active file could create too much prompt overhead if the UX is not designed carefully.
- The ship and document workflows need explicit ownership boundaries so they update the same consolidated file without duplicating or contradicting each other.

### Recommended Next Step
- Use `/research-astack` to inventory every place in the repo that assumes `00-scope.md` through `03-progress.md`, then map how each one should read from or write to the consolidated release-folder contract.

## Plan

### Summary
- Replace the repo-root numbered workflow artifact contract with a direct `docs/releases/` contract in one pass.
- Make this the real documentation versioning and archival process for astack itself.
- Standardize on `docs/releases/VERSION`, `docs/releases/RELEASE_LOG.md`, and `docs/releases/<version>-<slug>.md`.
- Require `/ship-astack` and `/document-release-astack` to convert the enhancement record from an in-flight document into a shipped, past-tense archive entry.

### What Already Exists
- The current astack workflow already has clear lifecycle stages and strong separation between source templates, generated outputs, docs, and tests.
- The repo already has versioning and changelog habits that can be adapted into the release-folder contract.

### Architecture
- `docs/releases/VERSION` stores the next enhancement-doc version for this repo's archival workflow.
- `docs/releases/RELEASE_LOG.md` stores the summarized cross-enhancement history.
- `docs/releases/<version>-<slug>.md` stores the complete enhancement lifecycle.

### Concrete Changes
- Update source workflow templates so they read and write consolidated-file sections instead of repo-root numbered files.
- Update downstream skills so they consume the consolidated release artifact rather than `02-plan.md` and `03-progress.md`.
- Update `scripts/gen-skill-docs.ts` to inject release-folder readiness rules, artifact resolution guidance, and ship/dashboard text for the new contract.
- Regenerate committed Claude, Codex, and Copilot skill outputs.
- Update public and contributor docs to remove numbered repo-root artifact guidance and describe the release-folder method as canonical.
- Update tests and fixtures to create `docs/releases/VERSION`, `docs/releases/RELEASE_LOG.md`, and versioned enhancement files.

### UX And States
- No enhancement file exists: ask for a slug, increment `docs/releases/VERSION`, create `docs/releases/<version>-<slug>.md`, and initialize the standard sections.
- Exactly one enhancement file exists: confirm the exact filename, then proceed.
- Multiple enhancement files exist: ask which file to use before writing.
- In-flight sections may be present/future tense.
- Shipped sections must be past tense.

### Implementation Phases
#### Phase 1: Lock the release-folder contract
- Define file names, section order, version ownership, and file-resolution rules.

#### Phase 2: Migrate downstream skills and ship/doc behavior
- Replace numbered-file reads with consolidated-section reads.
- Make `/ship-astack` update `docs/releases/VERSION`, append `docs/releases/RELEASE_LOG.md`, and mark the enhancement record as shipped.
- Make `/document-release-astack` rewrite enhancement prose into final archival past tense while syncing repo docs.

#### Phase 3: Regenerate outputs and update written docs
- Keep every committed skill artifact and repo doc aligned with the new contract.

#### Phase 4: Retarget tests and fixtures
- Replace numbered-file fixtures and literal assertions with release-folder fixtures and consolidated-file expectations.

### Failure Modes
- Some skills still expect numbered repo-root files.
- Ship readiness still depends on `02-plan.md`.
- Ship and document-release both rewrite the enhancement record differently.

### QA And Test Matrix
- Generated root, Claude, Codex, and Copilot skill docs all describe `docs/releases/VERSION`, `docs/releases/RELEASE_LOG.md`, and `docs/releases/<version>-<slug>.md`.
- Scope writes `Scope`.
- Research writes `Research`.
- Plan writes `Plan`.
- Implement writes `Progress`.
- Ship increments release-doc versioning and appends the release log.
- Document-release converts enhancement prose into past tense and aligns repo docs with shipped state.

### Rollout Notes
- This is a direct cutover with no compatibility shims.
- The release-folder method becomes the canonical archival/versioning model for this repo.

### Not In Scope
- Changing unrelated product-version automation outside the workflow documentation contract.
- Designing a separate UI for browsing archives.

### Open Questions
- Default assumption: ship updates structural status, version, and release-log entries first, and document-release performs the final prose normalization into archival past tense.
