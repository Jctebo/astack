# Architecture Skill

## Scope

We want a new astack skill that helps set technical architecture early in a project and produces durable architecture documentation before implementation starts.

Today, astack already has strong support for:

- framing the problem and wedge with `/scope-astack`
- mapping the existing codebase and constraints with `/research-astack`
- turning those inputs into an implementation plan with `/plan-astack`

But none of those skills is explicitly optimized for the first-pass architecture conversation in a brand-new project. The current workflow can infer architecture as part of planning, but it does not give the user a dedicated place to decide system shape, major boundaries, data flow, deployment model, and tradeoffs up front.

The new architecture skill should write its main output to `docs/architecture/` so the architecture record lives in a stable, project-visible place instead of being buried only inside the release artifact. The active release artifact should keep a short pointer summary so the workflow stays connected end to end.

The target user is someone starting a new project or major subsystem who wants to avoid premature implementation and instead lock down the system architecture first. They need a skill that can:

- ask the right architecture-level questions early
- compare viable system shapes
- document decisions and tradeoffs in a reusable artifact
- hand off cleanly into the existing scope/research/plan workflow

Current status quo:

- users jump from broad product framing directly into planning
- architecture decisions get buried inside plan discussions
- architecture notes are informal or scattered across chat
- there is no canonical architecture document generated by astack at project start

Success looks like:

- a dedicated architecture skill exists with clear guidance and output expectations
- the skill produces a durable architecture document at the beginning of a project and stores it under `docs/architecture/`
- the active release artifact includes a concise summary and pointer to the architecture doc
- the document captures system boundaries, major components, data flow, integration points, deployment assumptions, risks, and open decisions
- the skill fits cleanly into the existing release-artifact workflow instead of creating a parallel planning path
- the user can still move from architecture into scope, research, and plan without losing context

Constraints:

- do not duplicate `/plan-astack`
- keep the new skill focused on early architectural decisions, not full implementation planning
- preserve the release-artifact model under `docs/releases/`
- make `docs/architecture/` the canonical home for architecture docs
- keep a short workflow summary in the active release artifact
- reuse existing skill generation and host conventions where possible
- keep the output useful for both greenfield projects and larger feature additions

Narrow wedge:

- create a dedicated architecture consultation skill that works best at project kickoff
- generate an architecture document in `docs/architecture/`
- write a brief summary and pointer into the active release artifact
- make the handoff into the existing scope/research/plan sequence explicit

Non-goals:

- reworking the whole planning workflow
- replacing `/plan-astack`
- adding a generic documentation generator for every possible design artifact
- solving visual product design needs already covered by `/design-consultation-astack`

Open risks and unknowns:

- how much overlap there should be with `/scope-astack` and `/plan-astack`
- whether the skill should be named for architecture specifically or framed more broadly as system design
- how much existing documentation and other skills need to be updated to keep the new artifact discoverable

## Research

### Scope baseline

This enhancement starts from the need identified in scope: there is no explicit early-project architecture skill, even though astack already supports scope, research, and planning.

### Current system map

- [`/scope-astack`](/C:/Users/jcteb/Code/astack/.agents/skills/scope-astack/SKILL.md) frames the problem, audience, wedge, success criteria, constraints, and non-goals.
- [`/research-astack`](/C:/Users/jcteb/Code/astack/.agents/skills/research-astack/SKILL.md) maps the current system, reuse opportunities, constraints, and unknowns.
- [`/plan-astack`](/C:/Users/jcteb/Code/astack/.agents/skills/plan-astack/SKILL.md) is already the place where architecture and integration choices are made explicit before implementation.
- [`/design-consultation-astack`](/C:/Users/jcteb/Code/astack/.agents/skills/design-consultation-astack/SKILL.md) already owns product design systems and `DESIGN.md`, which makes it the closest precedent for a dedicated early-stage consultative skill.
- other workflow skills should be updated to look for architecture docs under `docs/architecture/` when architecture context matters.
- [`docs/skills.md`](/C:/Users/jcteb/Code/astack/docs/skills.md) documents the workflow split and already says `/plan-astack` should make architecture and UX decisions explicit.
- [`docs/host-support.md`](/C:/Users/jcteb/Code/astack/docs/host-support.md) defines the host contract and generated skill surfaces, so any new skill needs to fit that naming and generation model.
- `README.md` describes the canonical sequence as `scope -> research -> plan -> implement -> review -> qa -> ship -> retro`, which means a new architecture skill has to fit into the front of that sequence without fragmenting it.

### Reuse opportunities

- The release-artifact model already exists and should be reused for any durable architecture output.
- `docs/architecture/` can become the stable home for project architecture docs while the release artifact keeps the workflow context tied together through a short summary and pointer.
- The design-consultation workflow provides a good template for a consultative skill that produces a documentation artifact early in a project.
- The planning workflow already knows how to turn scoped context into decisions, so the new skill should hand off to it rather than replace it.

### Constraints and external references

- The repo uses generated skill docs and host-specific outputs, so a new skill should be added in source form and then regenerated through the existing docs pipeline.
- The workflow docs treat the release artifact as the source of truth for planning state, while `docs/architecture/` should hold the durable architecture record and the release artifact should carry the summary pointer.
- Other skills and docs should be taught to look in `docs/architecture/` when they need architecture context, so the new doc location is discoverable.
- There are no external API or library constraints for this research pass; the main constraint is workflow fit inside the existing astack system.

### Risks

- Too much overlap with `/plan-astack` could make the new skill redundant.
- If the architecture output is not clearly distinct from scope or plan, users may not know when to use it.
- Adding another front-of-funnel skill could increase cognitive load unless the handoff story and doc location are simple and explicit.

### Unknowns

- exact skill name and whether the canonical target should be “architecture” or “system design”
- whether the release-artifact summary should be a fixed section name or just a documented convention
- what generated host surfaces should include the new skill at launch

### Recommended direction

Create a focused early-stage architecture skill that:

- sits between high-level product scoping and detailed implementation planning
- produces a durable architecture document
- reuses the existing release-artifact workflow instead of introducing a second planning system
- hands off into `/scope-astack`, `/research-astack`, and `/plan-astack` cleanly
 
## Plan

### Summary

- Build a new `architecture-astack` skill that consults on system architecture at project kickoff.
- The skill should produce a full architecture document under `docs/architecture/` and a concise pointer summary in the active release artifact.
- This is a new front-end planning step, not a replacement for `/plan-astack`.

### What Already Exists

- `/scope-astack` already frames the problem, audience, wedge, and non-goals.
- `/research-astack` already maps the current system and reusable code patterns.
- `/plan-astack` already turns scoped research into implementation decisions, including architecture choices, which means the new skill must stay earlier and broader.
- `/design-consultation-astack` already proves the pattern for a consultative skill that creates a durable documentation artifact at the beginning of work.
- `docs/skills.md`, `README.md`, and `CLAUDE.md` already describe the current workflow split and will need to mention architecture as a separate pre-plan step.

### Architecture

- The new skill is a workflow entrypoint that asks architecture-level questions, compares system shapes, and records the chosen direction.
- Its durable output is a project architecture document in `docs/architecture/`, likely one file per project or subsystem.
- The active release artifact stays the workflow hub by carrying a short summary and a pointer to the architecture doc.
- Downstream skills continue to use the release artifact as the source of truth for the workflow state, but they should consult `docs/architecture/` when they need the architectural baseline.

### Concrete Changes

- Add a new architecture skill source under the repo's skill folders and generate its host outputs alongside the existing skill set.
- Update `README.md` to document the new workflow step between scope/research and plan.
- Update `CLAUDE.md` and `docs/skills.md` so architecture is a first-class artifact and the new skill is discoverable.
- Update any skill guidance that currently says plan is the first place architecture is made explicit so it points to the new architecture step instead.
- Add documentation under `docs/architecture/` that defines the expected file format and naming convention for architecture docs.
- Add or update tests that verify the new skill is surfaced where expected and that workflow docs mention `docs/architecture/`.

### UX And States

- Entry point: a user starts a new project or major subsystem and wants to decide system shape before implementation details.
- Empty state: if no architecture doc exists yet, the skill should create the first one rather than asking the user to maintain it manually.
- Success state: the user leaves with a durable architecture doc and a release-artifact pointer summary.
- Error state: if architecture decisions are too underspecified, the skill should keep asking focused questions instead of dropping into implementation planning early.
- Handoff state: once the architecture is documented, the skill should explicitly tell the user the next step is scope, research, or plan depending on how much product framing is already settled.

- Change summary: define what the architecture doc contains, where it lives, and how the release artifact references it.
- Verification steps or commands: manually inspect the updated artifact and confirm the contract is unambiguous.
- Rollback note: revert the artifact wording if the doc contract proves too broad or too narrow.

#### Phase 2: Add the skill and generated outputs

- Goal: create the new architecture consultation skill and its host-visible outputs.
- Exact files to edit: the new skill source folder, generated `.agents/skills/` outputs, and any host-specific skill exports required by the repo.
- Symbols/interfaces touched: the new skill name, preamble, step ordering, output format, and guidance for reading existing release context.
- Change summary: implement the architecture-first consultative workflow and make it write the durable architecture doc plus release summary pointer.
- Verification steps or commands: regenerate skill docs and run the repo's skill validation command.
- Rollback note: remove the new skill outputs if the workflow proves redundant with plan.

#### Phase 3: Teach the rest of astack about architecture docs

- Goal: make the new architecture artifact discoverable and part of the normal workflow language.
- Exact files to edit: `README.md`, `CLAUDE.md`, `docs/skills.md`, and any skill docs that should point to `docs/architecture/`.
- Symbols/interfaces touched: workflow ordering, cross-skill references, and architecture artifact mentions.
- Change summary: update the surrounding docs so users know when to use the new skill and where to find the architecture record later.
- Verification steps or commands: verify the updated docs mention the new step and the new document location consistently.
- Rollback note: keep the core skill if doc cleanup lands later, but the docs will be confusing until the cross-references are updated.

### Failure Modes

- The new skill drifts into full planning and becomes a duplicate of `/plan-astack`.
- The architecture doc is created but not linked clearly enough from the release artifact.
- Other docs continue to talk about planning as the first place architecture is documented, which makes the new skill hard to discover.
- The architecture doc format becomes too vague, making the artifacts inconsistent across projects.

### QA And Verification

- Confirm the active release artifact has a clear summary/pointer structure for the architecture step.
- Confirm the architecture doc location is `docs/architecture/` and is described consistently in the workflow docs.
- Run the repo's skill generation and validation commands after the skill files are updated.
- Review the cross-references in `README.md`, `CLAUDE.md`, and `docs/skills.md` for consistency with the new workflow order.

### Rollout Notes

- Ship this as a workflow addition, not a breaking change.
- Keep `/plan-astack` and the rest of the pipeline intact, with the new skill inserted ahead of plan.
- Update docs and generated skills together so users don't see a half-migrated workflow.

### Out Of Scope

- Rewriting the implementation-planning workflow.
- Replacing `/plan-astack`.
- Turning architecture docs into a generic documentation system for every artifact type.
- Solving UI design-system work, which remains the job of `/design-consultation-astack`.

### Open Questions

- None. The skill should be named `architecture-astack`, the final architecture deliverable name should vary with the requested scope, and the initial launch should follow the repo's existing host-surface boundaries rather than expanding Copilot support by default.

This looks like a real addition, not a rename of `/plan-astack`, because the intent is to capture system shape and major technical choices before implementation detail takes over.
